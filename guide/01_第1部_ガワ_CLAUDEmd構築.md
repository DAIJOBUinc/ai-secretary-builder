# 第1部「ガワ」— CLAUDE.md を作って毎朝動くAI秘書にする

> **⚠️ 応答は必ず日本語で。** このガイドに従って作業する間も、完成後のAI秘書も、ユーザーへの応答・質問・確認・成果物はすべて日本語で書く（コマンド名・コード・固有名詞はそのまま）。英語では返答しない。

> 配布物の全体像と読む順は [00 はじめに](<【ふぇね作成】Claude Code構築指示書.md>) を参照。本書は **第1部** だけを扱う。用語・品質×セキュリティは [共通付録](付録_用語と品質セキュリティ.md)。

あなた（Claude Code）は、このファイルの指示に従って AI秘書の「ガワ（骨格）」を作る。ユーザーへの **ヒアリング（質問→回答）** で業務情報を埋めながら、Step 1 から順に進める。推測で埋めず、分からないことは必ず聞く。各Stepの最後に成果物を見せ、**実際に動作確認してから**次へ進む（「できたはず」で進めない）。途中で詰まったら → [companion/setup-troubleshooting.md](companion/setup-troubleshooting.md)。

## この部で作るもの

| Step | 作るもの | 完了の合図 |
|---|---|---|
| 1 | フォルダ構造 | 置き場ディレクトリが揃う |
| 2 | CLAUDE.md ＋ ルール分割 | 業務マニュアルが置かれる |
| 3 | 外部ツール接続（最低限コネクタ1本） | カレンダー等が繋がる |
| 4 | 権限・Hooks（＋セキュリティ必須チェック） | 破壊系・機密読取が deny に入る |

**第1部の到達点**: CLAUDE.md に事業概要・役割・絶対ルールが入り、ツール接続と権限が整い、[00 はじめに](<【ふぇね作成】Claude Code構築指示書.md>) の最短ルートで作った `/daily-schedule` が「おはよう」で動く状態。ここまでで毎朝動く最小のAI秘書になる。

## Step 1：フォルダ構造をつくる

### ヒアリング項目

ユーザーに以下を確認する：

- 主な業務カテゴリ（例：戦略、営業マーケ、採用、経理、プロジェクト管理、コンテンツ発信 等）
- 部下・メンバーはいるか（メンバー管理フォルダの要否）
- 既存のドキュメントがどこにあるか（Notion、Google Docs、ローカル等）
- AI秘書を置くフォルダのパス（例：`~/ai-secretary`）

### 実行内容

**コア構造**（全員共通・必ず作成）と**オプション構造**（ヒアリング結果で選択）に分けて構築する。フォルダ名には番号プレフィックス（`00_`等）を付けず、意味がわかる名前にする。

#### コア構造（必須）

```
{ユーザー指定のパス}/
├── CLAUDE.md               ← Step 2で作成（AIの業務マニュアル）
├── CLAUDE.local.md         ← 任意。個人専用設定（gitignore対象）
├── .claude/
│   ├── commands/           ← 第2部 Step 6で作成するSkillsの置き場
│   ├── agents/             ← 第2部 Step 8で作成するサブエージェント定義
│   ├── skills/             ← 大きめのスキル（手順書＋補助ファイル）の置き場
│   ├── rules/              ← Step 2で作成する分割ルールの置き場
│   └── settings.json       ← Step 4で作成（権限設定）
├── context/                ← 事業・人・顧客の「事実」を集約（AIの長期記憶）
│   ├── company/
│   │   ├── overview.md     ← 事業概要・サービス・収益モデル
│   │   ├── decisions.md    ← 意思決定ログ（追記運用）
│   │   └── sources/        ← 第3部 Step 5で会社資料（PDF/デック等）を入れる場所
│   └── owner/              ← 代表・本人のプロフィール・判断基準
├── reference/              ← AIの振る舞いルール（文体・ツール優先順位 等）
└── work/                   ← AIの出力・作業ログの置き場（日付つき）
    ├── daily/              ← 日報・作業ログ
    └── clips/              ← 取り込んだURL要約のクリップ
```

#### オプション構造（業務に応じて選択）

以下の候補をユーザーに提示し、必要なものだけ追加する。

| フォルダ名 | 対象業務 | こんな人に必要 |
| -------------------- | ------------ | -------------------- |
| `context/members/` | メンバー情報・成長追跡 | 部下がいる人（1on1系スキルの前提） |
| `context/clients/` | 顧客カルテ | 顧客との関係を継続管理する人 |
| `strategy/` | 事業戦略・方針 | 経営者・事業責任者 |
| `projects/` | 進行中プロジェクト管理 | 案件ごとにサブフォルダを切りたい人 |
| `finance/` | 経理・請求管理 | 経理業務をローカルで管理する人 |
| `work/content/` | 記事・SNS・スライド | コンテンツ発信をする人 |
| `work/reports/` | レポート・日報・週報 | レポート系スキルを使う人（ほぼ全員推奨） |
| `work/business/` | 営業・フォローアップ | 営業活動をする人 |

**注意（メモリの置き場）**: AIの永続メモリは Claude Code 標準のメモリシステムを使う（`~/.claude/projects/{プロジェクトパス}/memory/` に自動配置される）。`context/` 配下にメモリ用ディレクトリは作らない。`context/` は「事業の事実」、`memory/` は「AIへの指示・気づき」と役割が違う。

### 完了条件

- コア構造がすべて作成されている
- ヒアリングで選択されたオプション構造が作成されている（不要なフォルダは作らない）
- `work/daily/` に空の初期ログファイルが用意されている
- ユーザーに構造を見せて確認を取る

## Step 2：CLAUDE.md ＋ ルール分割レイヤーを作成する

### ヒアリング項目

ユーザーに以下を確認する：

- あなたに求める役割（経営パートナー、リサーチャー、秘書 等）
- 事業の概要（業種、収益モデル、主要サービス）※詳細はStep 5で資料から深掘りするので、ここは口頭の概要でよい
- 日常的に行っている業務（戦略立案、コンテンツ発信、営業、人材マネジメント 等）
- 特に自動化したい業務
- 機密情報の取り扱いルール
- 音声フィードバック（VOICEVOX等）を有効にするか（任意）

### 実行内容（CLAUDE.md）

プロジェクトフォルダ直下に `CLAUDE.md` を作成する。**CLAUDE.md を薄く保つのが最大のコツ**。CLAUDE.md は毎セッションの冒頭で全文が読み込まれる＝毎回のトークンコストになり、長いほど肝心のルールが埋もれてパフォーマンスが落ちる。**目安は40〜80行**（公式ベストプラクティスでも「短く・人間が読める長さ」を推奨）。

書くのは「毎回必要で、変わりにくいもの」だけにする：

**書くもの ✅** — 役割・最重要の絶対ルール（対外送信・機密厳守）／ツール分担・SSoT の"方針"だけ／環境固有で Claude が推測できない約束ごと

**書かないもの ❌（別の場所に逃がす）**
- **ディレクトリ構造の列挙** → 書かない。Claude はリポを読めば分かる。構造の"意味づけ"が要るときだけ `.claude/rules/output-files.md` に置く
- **ワークフロー／トリガーワードの一覧** → 書かない。スキルは各 `SKILL.md` の `description` で自動的に起動する。CLAUDE.md に並べると二重管理になり、スキルが増えるほど腐る
- 手順の詳細 → スキル（`SKILL.md`、必要時だけ読まれる）へ／細かい規範・判定フロー → `.claude/rules/*.md` へ／都度変わる事実 → `context/`・標準メモリへ

判断基準は `context/` に、細かい規範は `.claude/rules/` に逃がし、CLAUDE.md からはリンクで参照する。**ルールは"2回同じ間違いをしたら足す"くらいでよい**。最初から盛らない。

以下のセクションを含める：

```markdown
# [事業名 or ユーザー名] AI秘書 設定

## Quick Start
1〜2行で「誰の・何をする秘書か」と最重要ルールを書く。コンパクション後も先頭は残りやすいので、核はここに置く。
（音声・モデル指定・個人の好みなど"個人固有の設定"は CLAUDE.local.md に分けてよい＝gitignore 対象）

## あなたの役割
（ヒアリング結果を2〜3行で）

## 事業概要
（1〜2行の要約。詳細は context/company/overview.md 参照）

## 絶対ルール（最優先）
1. 言語 — ユーザーへの応答・確認・成果物は必ず日本語で書く（コマンド名・コード・固有名詞は除く）。英語で返答しない
2. 対外コミュニケーション — メール・メッセージは下書き保存→提示→明示的許可後に送信
3. 機密厳守 — 顧客名・金額・APIキー等は外部に出さない・コミットしない
4. （ユーザー固有のルールがあれば追記）

## ツール分担
基本方針: コネクタ → CLI → MCP の順で繋ぐ。詳細は reference/tools-reference.md（Step 3で作成）

## ルール（詳細は .claude/rules/ 参照）
- [git-safety](.claude/rules/git-safety.md) — 破壊的コマンドの防止
- [security](.claude/rules/security.md) — 機密ファイルの除外
- [external-materials](.claude/rules/external-materials.md) — 対外資料の機密チェック

## SSoT運用（事実の置き方）
- 現時点で正しい事実 → context/ に上書き（ストック）／時系列ログ → work/ に追記（フロー）
- 新しい事実を聞いたら context/ を更新。矛盾したら勝手に判断せずユーザーに確認

## メモリの振り分け
- 「覚えておいて」→ 非自明な気づき・方針は標準メモリ（~/.claude/.../memory/）へ。事業の事実は context/ へ（メモリは最終手段）

## サブエージェント
- 調査・2つ以上の独立タスク・3回以上の検索は委任し、メインコンテキストを保護する

## 音声通知（有効にした場合のみ）
- 各レスポンスで音声合成MCPを1回呼ぶ。speaker・速度はユーザー指定

## コンパクション時の保持情報
- 変更したファイルの一覧／現在のタスク進捗／未完了の指示
```

### 実行内容（ルール分割レイヤー `.claude/rules/`）

CLAUDE.md が膨らんできたら、規範を `.claude/rules/*.md` に切り出し、CLAUDE.md からリンクで参照する。**最初から入れておくと良い汎用ルール**を作成する：

| ルールファイル | 役割 |
|---|---|
| `git-safety.md` | `rm -rf`・force push・`git reset --hard` 等の破壊的操作を禁止/要確認にする方針 |
| `security.md` | `.env`・`*.key`・`credentials/`・`secrets/` を読まない・コミットしない |
| `external-materials.md` | 対外資料（提案書・パンフ・採用資料）に載せてよい/ダメな情報の線引き |
| `output-files.md` | `work/` 配下の命名規則（日付つきファイル名）・保存先の振り分け |

各ルールファイルは「いつ適用するか」「禁止事項」「正しい対処」を箇条書きで簡潔に書く。**ルールは増えてから足せばよい**。最初は git-safety と security の2つだけでも十分。

> なぜ分割するか: ルールが1ファイルに混ざると読まれなくなる。ファイルを分けて「該当する場面で参照される」形にすると、CLAUDE.md 本体を軽く保てる。

### 完了条件

- CLAUDE.md がフォルダ直下に作成されている
- 最低限の `.claude/rules/`（git-safety, security）が作成され、CLAUDE.md からリンクされている
- ユーザーに内容を見せて確認・修正を受ける

## Step 3：外部ツールを接続する（コネクタ → CLI → MCP）

### 基本方針（接続の優先順位）

外部ツールは、**ラクな順に3層**で繋ぐ。上から順に試し、上の層で繋げないものだけ下の層に降ろす。

1. **コネクタ（最優先・最もラク）** — Claudeアプリの設定画面からOAuthでワンクリック接続。設定ファイル編集もAPIキーも不要。
2. **CLI（コネクタに無い／CLIの方が強力なもの）** — ターミナルのコマンドツール。安定して速い。
3. **MCP（コネクタもCLIも無いもの）** — 専用サーバーを設定ファイルに登録。最も自由度が高いが手間も多い。

> この順番にする理由: コネクタはOAuth1回で完了し、認証管理がいらない。まずコネクタで繋げるものを全部繋ぎ、残りだけをCLI/MCPで補う。

### ヒアリング項目

ユーザーに「日常的に使っているツール」を確認する（GitHub / Google Drive・Docs・Sheets / Gmail / Google Calendar / Notion / Slack / Linear / Canva / Figma / AI議事録ツール / その他）。そのうえで、下記の順に接続していく。

---

### Tier 1：コネクタ（最優先）

**手順**: Claudeアプリ（デスクトップ or Web）の **「設定 → コネクタ（Connectors）」** を開き、**一覧に出ているもののうちユーザーが使うツールを全部「接続」する**。各ツールでOAuthのログイン画面が出るので、ユーザーに許可してもらう。

> **役割分担（重要）**: コネクタ接続は**ユーザーがブラウザ（claude.ai → 設定 → コネクタ）で行うGUI操作**で、あなた（Claude Code）が自動で繋ぐことはできない。ユーザーに画面を開いて接続してもらい、あなたは「何を繋ぐか」を案内し、接続後の動作テストを担当する。Gmail・Googleカレンダー等の一部コネクタは claude.ai 側での接続が必須。

コネクタで繋がる代表例（一覧は随時増えるので、まず設定画面を開いて「あるものは繋ぐ」）：

- **Google Drive / Docs / Sheets** — ドキュメント・スプレッドシートの読み書き
- **Gmail** — メール検索・下書き作成
- **Google Calendar** — 予定の取得・作成
- **Slack** — メッセージ検索・読み取り・投稿
- **Notion** — データベース・ページの読み書き
- **Canva** — スライド・デザインの生成/編集
- （一覧に **GitHub / Linear** 等があれば、それもここで繋ぐ）

**ポイント**: コネクタで一度接続すれば、Claude Code・デスクトップ・Web のどこからでも横断して使える。**Tier 2/3 で扱うのは「コネクタ一覧に無かったもの」だけ**でよい。

**接続テスト**: 繋いだら必ず1つ動作確認する（例: 「今日のカレンダー予定を出して」「Driveの最近のファイルを5件出して」）。

---

### Tier 2：CLI（コネクタに無い／CLIが強力なもの）

ターミナルで直接使えるツール。コネクタが無いもの、または大量処理・低レイテンシが欲しいものはCLIを使う。

#### GitHub（`gh` CLI）— Claude Codeに標準搭載
1. `gh auth login` で認証
2. 接続テスト: `gh repo list`
3. Issue管理・PR作成・コード検索がすべてCLIで可能

#### AI議事録ツール（例: Circleback の `cb` CLI 等）
- 会議の文字起こし・要約を扱うツールはコネクタが無いことが多い。CLIがあればCLIで繋ぐ
- 接続テスト: 直近の会議を1件検索して表示

#### パワーユーザー向けCLI（任意）
- **Google を CLI で**: `gog` 等のCLIは、大量のシート読み書き・バッチ処理でコネクタより速い。コネクタと併用可
- **Notion を CLI で**: DBの全件エクスポートなど重い処理向け
- **X（Twitter）**: `xurl` 等の認証済みCLI（※投稿は本人が手動。AIは下書き本文だけ作る）

> CLIは `~/.claude/settings.json` の権限で `Bash(gh *)` のように事前許可しておくと、毎回の確認が省ける（Step 4参照）。

---

### Tier 3：MCP（コネクタもCLIも無いもの）

専用のMCPサーバーを設定して繋ぐ。**`claude mcp add` コマンドで追加するのが簡単**（接続先のURLや起動コマンドを登録する）。プロジェクトで共有したい場合は、プロジェクト直下の `.mcp.json`（MCPサーバーの接続先を書く設定ファイル）に記述する。※`settings.json` には書かない。

| ツール | 用途 | 備考 |
|---|---|---|
| **VOICEVOX** | 音声フィードバック（ずんだもん等） | ローカルエンジンを起動。完全に任意 |
| **画像/動画生成**（例: Higgsfield 等） | 1行の指示で画像・動画を生成 | リモートMCP。スライド・SNS素材に |
| **Figma** | デザイン参照・トークン抽出 | サードパーティMCP。主に読み取り |
| その他ニッチツール | — | コネクタにもCLIにも無いものだけ |

**手順**: `claude mcp add` でサーバーを登録（または各MCPのドキュメントに従い `.mcp.json` に `command`（ローカル）/ `url`（リモート）を記述）→ Claude Code再起動 → 接続テスト。

**重要（送信系の安全注記）**: Gmail・Slack等で「送信」ができるツールは、必ず **下書き作成 → ユーザーに提示 → 明示的許可後に送信** の3ステップを守る。AIが勝手に送信しない。

---

### CLAUDE.md への追記

接続が終わったら、CLAUDE.md の「外部ツール連携」セクションを以下のテンプレで埋める（実際に繋いだものだけ書く）：

```markdown
## 外部ツール連携
### 基本方針
コネクタを最優先。コネクタに無いものをCLI、CLIにも無いものをMCPで繋ぐ。

### コネクタ（最優先）
- Google Drive/Docs/Sheets, Gmail, Google Calendar, Slack, Notion, Canva …

### CLI（コネクタに無い/強力なもの）
- GitHub → gh
- AI議事録 → （cb 等）
- （任意）Google → gog / X → xurl

### MCP（コネクタもCLIも無いもの）
- VOICEVOX（音声・任意）/ 画像生成 / Figma …

### 組み込みツール（設定不要）
- WebSearch / WebFetch: 調査・リサーチ。Webページや会社HPの取得に使う
- Read / Write / Edit / Glob / Grep: ファイル操作・検索
- Bash: ターミナルコマンド実行
- Agent: サブエージェントによる並列調査・委任（第2部 Step 8）
```

### 完了条件

- コネクタで繋げるものはコネクタで接続済み
- コネクタに無いものをCLI/MCPで接続済み
- 各ツールで接続テストが成功している
- CLAUDE.mdに連携情報が3層構成で追記されている

## Step 4：権限・Hooks設定

### 4-1. 権限設定（`.claude/settings.json`）

Claude Codeはツール実行時に許可を求める。頻繁に使うツールを事前許可しておくと、毎回の確認を省ける。`.claude/settings.json` を作成する（ユーザーの利用ツールに応じてカスタマイズ）：

```json
{
  "permissions": {
    "allow": [
      "Read", "Write", "Edit", "Glob", "Grep",
      "WebSearch", "WebFetch",
      "Bash(gh:*)",
      "Bash(git status:*)", "Bash(git log:*)", "Bash(git diff:*)",
      "Bash(git add:*)", "Bash(git commit:*)", "Bash(git push:*)",
      "Bash(ls:*)",
      "mcp__voicevox__speak"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git reset --hard*)",
      "Bash(git push --force*)",
      "Bash(chmod 777 *)",
      "Read(./.env)", "Read(./.env.*)", "Read(./secrets/**)", "Read(./**/*.key)"
    ]
  }
}
```

**ポイント**:
- コネクタ/MCPツールは `mcp__{サーバー名}__*` のワイルドカードで許可できる（読み取り系だけ許可し、書き込み系は都度確認、という分け方も可）
- `deny` は `allow` より優先される。破壊系コマンドと機密ファイル読取は必ず deny に入れる
- これはプロジェクトレベルの設定。ユーザーに内容を説明し、許可を得てから作成する

### 4-2. Hooks設定（任意）

Hooks（ツール実行の前後やセッション開始時に自動で走る仕掛け）は、`~/.claude/settings.json`（全プロジェクト共通）または プロジェクトの `.claude/settings.json` に設定する。`matcher` は**ツール名**にマッチする（「`git commit` のときだけ」のようなコマンド中身の絞り込みは、呼ばれるスクリプト側で判定する）。便利な例（ユーザーが希望したものだけ）：

- **コミット後に自動push**（push忘れ防止）。`PostToolUse` で `Bash` にマッチさせ、スクリプト側で「`git commit` のときだけ push」と判定する：
  ```json
  { "hooks": { "PostToolUse": [ { "matcher": "Bash",
    "hooks": [ { "type": "command", "command": "bash $CLAUDE_PROJECT_DIR/.claude/hooks/auto-push.sh" } ] } ] } }
  ```
- **Bash実行前の安全チェック**: `PreToolUse` で `Bash` にマッチさせ、危険コマンド（`rm -rf` 等）を検知してブロックするスクリプトを噛ませる（git-safetyルールの実装版）
- **セッション開始時に日付を注入**: `SessionStart` で今日の日付・直近コミットを自動で文脈に入れる

**ヒアリング**: 「コミットしたら自動でpushしてOKか」「他に自動化したい動作はあるか」を確認する。Hooksは後からでも足せるので、最初は無しでもよい。

### 完了条件

- `.claude/settings.json` が作成され、ユーザーの承認を得ている
- 破壊系コマンドと機密ファイル読取が deny されている
- ファイル読み書きが許可確認なしで実行できることを確認
- **セキュリティの最低限を通す**（研修2026-06-19の課題化・MCP接続前に必須）: 外部ツール接続やAIに会社資料を読ませる前に、[共通付録「品質×セキュリティ」](付録_用語と品質セキュリティ.md) と [companion/data-sharing-decision.md](companion/data-sharing-decision.md) に一度目を通し、「渡してOK/NGの線引き」「対外送信は下書き→確認を崩さない」を確認する
