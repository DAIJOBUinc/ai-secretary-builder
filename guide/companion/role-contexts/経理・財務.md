# 経理・財務 向け スターターコンテキスト

> この役割の人がAI秘書を最速で戦力化するための初期セット。本体ガイド Step5（コンテキスト注入）から参照される。まずここから埋め、慣れたら広げる。
>
> 使い方：あなた（Claude Code）は、経理・財務担当であるユーザーに以下を順にヒアリングし、`context/` を下書きしていく。先に勘定科目表・請求/契約フォーマット・取引先一覧で埋められる所を埋め、足りない分だけ聞く。

## 1. まず入れる context/ ファイル（最初に用意する自社情報）

- `context/finance/accounts.md` — 勘定科目・仕訳ルール（経費区分・按分・税区分の判断基準）
- `context/finance/vendors.md` — 取引先・支払先一覧（請求/支払サイクル・条件・担当。機密度高）
- `context/finance/templates.md` — 請求書・契約フォーマットへの索引（定型文書の生成元）
- `context/finance/calendar.md` — 月次・締め日・支払サイト（処理の漏れと遅延を防ぐ運用カレンダー）

> 補足：締め日・サイクル・科目ルールはストック（上書き運用）として最新版だけを残す。日々の処理ログはフローとして追記。金額・口座などの数字は機密度が高く、`context/` には入れてよいが社外送信物には出さない。

## 2. 最初に渡すと効く自社資料（context/company/sources/ に置く）

- 勘定科目表・経費区分の基準 — 仕訳・分類の相談ができる（数字は社外秘）
- 請求書・契約書の定型フォーマット — 定型文書ドラフトの素材になる
- 取引先・支払先一覧（サイクル・条件付き） — 請求/支払の整理と抜け漏れ防止に効く
- 月次PL・予算 — 予実の相談ができる（いずれも機密度高、社外向けは要確認）

> 上記はいずれも機密度が高い。社外向けに出す前は必ず要確認。

## 3. 鉄板スキル（universal ＋ この役割の初回セット）

**universal（全員の土台・まず作る2つ）**
1. `/daily-schedule` — 朝、カレンダーとタスクから15分刻みの工程表を作る。締め日・支払・請求処理を漏らさず組む
2. `/mtg-prep` — 社内定例・取引先MTG前に、関連する条件・前回決定・関連資料を集めて準備資料を作る

**経理・財務の鉄板（universal に足す）**
3. `/daily-log` — その日の処理・対応をまとめて記録する。月次の振り返りと引き継ぎの土台になる
4. `/clip` — 税制改正・会計基準・補助金などの制度変更を要約・タグ付けして溜める（制度ナレッジの原料置き場）

**次に足す（優先順）**: 定型文書ドラフト系（請求・督促・稟議を自社ひな型で）→ 月次まとめ系（予実・資金繰りサマリ）

## 4. 経理・財務の Agent Team テンプレ — 月次まとめチーム

月次決算・予実分析など「集めて・分類して・まとめて・突き合わせる」重い仕事は、**分担チーム**が速い。`.claude/agents/` に定義し、スキルから起動する。

| エージェント | 役割 |
| --- | --- |
| 数値収集 | 各システム・台帳から対象期間の数値を集約 |
| 分類 | 勘定科目・部門・区分に沿って仕分け |
| サマリ作成 | 予実・資金繰りのサマリを作成 |
| 整合チェック | 元帳・証憑との突合、前月差異の点検 |

> フロー：数値収集→分類→サマリ作成→整合チェック。初回は単発の数値整理・要約から。慣れたらこのチームを組む（本体 第2部 Step 8 参照）。金額・取引先情報の扱いに注意（下の注意参照）。

## 5. この役割ならではの注意（Gotcha）

- **金額・口座・取引先条件は機密**：請求・支払・契約の数字、口座番号、取引条件、取引先別の金額は最高機密。宛先を間違えた相手や対外文書に出さない、`context/` に入れても外部送信物には出さない、リポジトリにコミットしない。送信系（メール・Slack）は下書き→提示→許可後に送信の3ステップを守る。
- **AI生成の数値は必ず元帳・証憑に当ててから確定**：合計・按分・予実差異などをAIに出させたら、推測値・概算値を確定値として扱わない。必ず元帳・証憑・原データに突き合わせてから確定する。端数処理・税区分・期ズレはAIが間違えやすいので、数字は人間が最終チェックする。
