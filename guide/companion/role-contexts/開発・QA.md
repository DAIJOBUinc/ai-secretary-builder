# 開発・QA 向け スターターコンテキスト

> この役割の人がAI秘書を最速で戦力化するための初期セット。本体ガイド Step5（コンテキスト注入）から参照される。まずここから埋め、慣れたら広げる。
>
> 使い方：あなた（Claude Code）は、開発・QAであるユーザーに以下を順にヒアリングし、`context/` を下書きしていく。先にリポジトリ・仕様書・テスト規約で埋められる所を埋め、足りない分だけ聞く。

## 1. まず入れる context/ ファイル（最初に用意する自社情報）

- `context/product/overview.md` — プロダクト概要・主要機能・対象ユーザー（仕様理解とテスト観点の土台）
- `context/product/architecture.md` — システム構成・技術スタック・主要コンポーネントの責務
- `context/dev/conventions.md` — コーディング規約・レビュー観点・ブランチ/PR運用ルール
- `context/qa/test_viewpoints.md` — 自社のテスト観点マスタ（機能/境界値/異常系/非機能などの抜けを防ぐ基準）
- `context/dev/glossary.md` — ドメイン用語集（仕様・Issue・テストで言葉の揺れを防ぐ）

> 補足：規約・観点マスタはLogic（規範）なので上書きで最新を保つ。設計判断の経緯は `context/company/decisions.md`（追記）に残すと、後でレビュー時に踏まえられる。

## 2. 最初に渡すと効く自社資料（context/company/sources/ に置く）

- 対象リポジトリ（または主要モジュールの説明） — コードレビュー補助・Issue整理の土台になる
- 仕様書・要件定義・設計ドキュメント — テスト観点/テストケース生成の素材になる
- 既存のテスト計画・テストケース・観点表 — 自社フォーマットに沿った生成ができる
- コーディング規約・レビューチェックリスト — レビュー指摘の基準が揃う
- 過去のバグ報告・障害レポート — 「やられやすい所」を学習させ、観点の精度が上がる

## 3. 鉄板スキル（universal ＋ この役割の初回セット）

**universal（全員の土台・まず作る2つ）**
1. `/daily-schedule` — 朝、カレンダーとタスクから15分刻みの工程表を作る。レビュー・実装・テスト設計をブロックで組む
2. `/mtg-prep` — 仕様会議・スプリントMTG前に、関連Issue・前回決定・仕様の論点を整理

**開発・QAの鉄板（universal に足す）**
3. `/interview` — 仕様・施策を実装着手前にAskUserQuestionで深掘りし、別セッションで実装できる spec に落とす（手戻り削減）
4. `/clip` — 技術記事・ライブラリ更新・脆弱性情報を要約・タグ付けして溜める（技術ナレッジの原料置き場）
5. `/daily-log` — その日のコミット・PR・Issue活動をまとめて記録する（進捗の可視化と引き継ぎ）

**次に足す（優先順）**: テスト系スキル（仕様書→テスト観点→テストケース。自社の観点マスタに合わせる）

## 4. 開発・QAの Agent Team テンプレ — テスト設計チーム

仕様からのテスト設計は、**分担チーム**で観点の網羅性が上がる。`.claude/agents/` に定義し、スキルから起動する。

| エージェント | 役割 |
| --- | --- |
| 仕様理解 | 仕様書・コードから機能と前提を把握 |
| テスト観点抽出 | 自社の観点マスタに沿って観点を洗い出す（機能/境界値/異常系/非機能/権限） |
| テストケース生成 | 観点をテストケースに展開 |
| レビュー | 網羅性の穴・過去バグ傾向との突合を指摘 |

> 観点は必ず `context/qa/test_viewpoints.md` のマスタを通す（下の注意参照）。初回は単発の仕様要約・Issue整理から（本体 第2部 Step 8 参照）。

## 5. この役割ならではの注意（Gotcha）

- **テスト観点は「自社マスタ」を必ず通す**：仕様書だけからテストケースを出させると、非機能・境界値・異常系・権限まわりが抜けやすい。`context/qa/test_viewpoints.md` の観点マスタを先に参照させ、網羅性を担保する。机上で観点を創作させず、過去のバグ傾向を踏まえさせる。
- **コード・認証情報・脆弱性詳細を外に出さない**：レビューや調査で触れるソースコード・APIキー・接続情報・発見した脆弱性の詳細は機密。`.env`・鍵ファイルは読ませない/コミットしない（Step4の deny 設定とpre-commitフックで二重に防ぐ）。脆弱性の具体的な再現手順や顧客システムの構成を対外資料・Issueの公開コメントに書かない。
