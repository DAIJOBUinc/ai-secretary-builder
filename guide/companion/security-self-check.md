# セキュリティ自己診断（10項目）

> これは「**自分のAI秘書が安全か**」の10秒セルフ診断。本体ガイド Step4（権限・Hooks）・Step10（Git安全）から参照される。
> **1つでも ✗ が残っていたら、直してから運用に乗せる**。各項目に「✅何を確認 / なぜ / 直し方」を付けてある。
>
> AI秘書を作る側（Claude Code）は、構築の最後にこのリストをユーザーと一緒に上から通し、✗ をその場で潰してから完了報告する。
> なお、これは「**作って終わりにせず、品質とセキュリティを担保してから渡す**」という体験版でもある（テスト設計・脆弱性診断の発想を、自分のAI環境に向ける）。

---

- [ ] **1. `deny` に破壊系コマンドと機密ファイル読取が入っているか**
  - ✅確認：`.claude/settings.json` の `permissions.deny` に `rm -rf` 等の破壊系、`.env`・鍵ファイルの読取が登録されているか
  - なぜ：許可漏れだと、AIが誤って消す・機密を読み上げる事故が起きうる
  - 直し方：`deny` に `"Bash(rm:*)"`・`"Read(.env)"`・`"Read(*.key)"` 等を追加（本体 Step4 のサンプル参照）

- [ ] **2. `.gitignore` に機密ファイルが入っているか**
  - ✅確認：`.gitignore` に `.env`・`*.key`・`*.pem`・`secrets/` 等があるか
  - なぜ：これが無いと鍵や認証情報がコミット＝履歴に残り、共有時に漏れる
  - 直し方：上記パターンを `.gitignore` に追記。既にコミット済みなら履歴からの除去も検討

- [ ] **3. 機密（顧客名・金額・個人情報）を `context/` に生で入れていないか**
  - ✅確認：`context/` 配下に契約金額・個人情報・顧客実名が平文で置かれていないか
  - なぜ：`context/` はAIが常時読む場所。Git管理する場合は流出面が一気に広がる
  - 直し方：機密度の高いものは別管理 or `.gitignore` 対象へ。対外物には出さない（→ [data-sharing-decision.md](data-sharing-decision.md)）

- [ ] **4. APIキー・トークンが `.env` 等に分離され、コミットされていないか**
  - ✅確認：キー類がソースやMarkdownに直書きされていないか
  - なぜ：直書きキーは流出の典型。1度コミットすると履歴に残り続ける
  - 直し方：`.env`・OS資格情報ストア等に分離し、コードからは環境変数で参照

- [ ] **5. 秘密スキャン（pre-commit等）が有効か ※Git管理する場合**
  - ✅確認：コミット前にキー・鍵・トークンを検知して止める仕組みが入っているか
  - なぜ：人間の目視は漏れる。最後の自動ガードがあると事故が激減する
  - 直し方：pre-commitフックを導入（本体 Step10）。Git管理しないなら本項目はスキップ可

- [ ] **6. 対外送信（メール/チャット）が「下書き→確認→許可」運用か**
  - ✅確認：CLAUDE.md に「外部送信は下書き保存→提示→明示許可後に送信」が明記され、実際にその挙動か
  - なぜ：誤送信・機密入りメールの自動送信は取り返しがつかない
  - 直し方：CLAUDE.md の絶対ルールに明記。AIが勝手に送る挙動を見たら即修正

- [ ] **7. `allow` が広すぎないか**
  - ✅確認：`permissions.allow` に不要なワイルドカード（例 `"Bash(*)"`）が無いか
  - なぜ：許可が広いほど、誤操作・破壊系がノーチェックで通ってしまう
  - 直し方：頻用するものだけ列挙（例 `"Bash(gh:*)"`・`"Read"`・`"Edit"`）。広すぎる許可は外す

- [ ] **8. 外部サービス（Notion等）で削除・破壊操作を許していないか**
  - ✅確認：接続したコネクタ/MCPに、削除・上書き・ゴミ箱操作の権限を渡していないか
  - なぜ：AIが本番データを消す・壊すと復旧コストが大きい
  - 直し方：可能なら読取中心の権限に絞る。破壊操作は人間が手で行う前提に。ルールにも明記

- [ ] **9. 出力ログ・`work/` に機密が漏れていないか**
  - ✅確認：`work/` 配下の調査結果・日報・下書きに、外に出せない機密が平文で残っていないか
  - なぜ：出力物は見落とされがち。Git管理だとそのまま履歴に乗る
  - 直し方：機密を含む出力は置き場と公開範囲を分ける。Git管理するなら `.gitignore` 対象を検討

- [ ] **10. 共有リポなら、個人情報・機密が分離されているか**
  - ✅確認：チームで共有するリポに、個人情報・自社機密が混ざっていないか
  - なぜ：共有範囲が広いほど、1ファイルの漏れが多人数に波及する
  - 直し方：機密は別リポ/別管理に分離、もしくは `.gitignore` で除外。共有前に中身を必ず一度確認
