# この情報、AIに渡してOK?（1ページ判断ツール）

> 「この資料、AIに食わせていいんだっけ？」で手が止まったとき用の1ページ判断ツール。本体ガイド 付録C（品質×セキュリティ）の実務版。
> **原則：判断に迷ったら渡さない。** 渡すのは後からでもできるが、漏れたものは取り戻せない。
>
> AI秘書を作る側（Claude Code）は、ユーザーが機密の扱いに迷ったらこのツリーで一緒に切り分ける。

---

## 判断ツリー（上から順に当てはめる）

```
渡したい情報がある
│
├─ それは APIキー / パスワード / 秘密鍵 / アクセストークンか？
│     └─ YES → ❌ NG。渡さない。.env 等に分離し .gitignore ＋ 秘密スキャンでガード
│
├─ それは 顧客機密 / 個人情報 / 契約金額 / 脆弱性情報か？
│     └─ YES → ⚠️ 要注意。下の「条件つきOK」を満たすときだけ context/ に入れる
│              満たせない / 自信がない → 渡さない
│
├─ それは 社内の一般情報（手順・自分の業務データ・社内資料）か？
│     └─ YES → ✅ OK。適切な権限下で context/ や work/ に入れて活用
│
├─ それは すでに公開されている情報（HP・公開資料・プレスリリース）か？
│     └─ YES → ✅ OK。そのまま渡してよい
│
└─ どれにも当てはまらない / 判断がつかない
      └─ ⚠️ 渡さない。1人で決めず確認する（迷ったら渡さない原則）
```

---

## 3分類と具体例

### ✅ OK（そのまま渡してよい）

- 公開情報：自社HP、公開済みのプレスリリース・登壇資料、一般に出回る業界情報
- 社内の一般情報：定型の業務手順、社内向け一般資料、自分が担当する業務データ
- 判断のしかた：「外部の誰かに見られても困らない」なら OK 寄り

### ⚠️ 要注意（条件つきで渡してよい）

- 対象：顧客機密、個人情報、契約金額・人月などの取引条件、社外秘の戦略・数値
- 渡してよい条件（すべて満たすこと）：
  1. 置き場の権限・公開範囲を確認した（Git管理なら `.gitignore` を確認）
  2. **対外資料・メール・チャットには出さない**ことを徹底できる
  3. 「いつの・誰の情報か」が分かっていて、抽象化の要否を判断できる
- 1つでも怪しければ → 渡さない、または社名・金額を伏せて抽象化してから渡す

### ❌ NG（渡さない）

- 対象：APIキー、パスワード、秘密鍵、アクセストークン、診断で見つけた具体的な脆弱性
- 扱い：`.env` 等に分離 → `.gitignore` で除外 → 秘密スキャン（pre-commit）で二重ガード
- これらは「便利だから」で渡してよくなることはない。例外なし

---

## 渡したあとの3つの約束

1. 機密を入れたら、その置き場が**Git管理か／共有範囲はどこか**を必ず確認する
2. 対外物（メール・チャット・提案資料）を作るときは、機密が混ざっていないか出す前に点検する
3. 迷いが残るなら、入れない方に倒す。**渡さなくて遅れることはあっても、漏れて困ることは無い**
