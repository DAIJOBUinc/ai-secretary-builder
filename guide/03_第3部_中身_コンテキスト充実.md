## Step 5：会社固有コンテキストの注入（最重要）

**AI秘書の質は、与えたコンテキスト（自社情報）の量と質でほぼ決まる。** ここが最も重要なステップ。一般論しか知らないAIを、「自社のことを分かっている秘書」に変える工程。

やることは4つ：**① 先に会社資料・HPを共有してもらう → ② AIが分かる範囲を context/ に下書きする → ③ ユーザーが確認・修正 → ④ 足りない情報をヒアリングで追記**（②〜④は繰り返す）。白紙のヒアリングシートをいきなり渡すのではなく、**先に資料で埋められるところを埋めてしまう**のがコツ。ユーザーは「ゼロから答える」より「下書きを直す」ほうが速くて正確。

### ① 先に会社資料・HPを共有してもらう（材料が先）

**白紙のヒアリングシートを渡す前に、まず手元の会社資料を集める**。先に材料があれば、AIが分かる範囲を埋められる。ユーザーに、**手元にある会社資料を `context/company/sources/` に入れてもらう**。具体的には：

- 会社案内・コーポレートパンフ（PDF）
- 営業資料・提案デック（PDF / PowerPoint）
- 価格表・サービス説明資料
- 会社HPのURL（ファイルでなくURLでよい）
- 既存の社内ドキュメント（Notion / Google Docs のリンクや書き出し）
- 過去の議事録・商談メモ

**AIはこれらをそのまま読める**：
- **PDF / Word / Excel / PowerPoint** → 対応スキル（`pdf`・`docx`・`xlsx`・`pptx` 等）や `Read` で中身を抽出できる。対応スキルが無ければ、内容をテキストで貼ってもらう
- **会社HP（URL）** → `WebFetch` で取得
- **Google Drive のファイル** → コネクタ、または Drive連携CLIで取得

資料が少なくても、**HPのURL 1本でも十分スタートできる**。まずは出せるものを出してもらう。

### ② AIが資料から context/ に下書きする（分かる範囲を埋める）

集めた材料を読み込み、**分かる事実を `context/` の適切な場所に振り分けて書き出す**。この時点では完璧でなくてよい。資料から読み取れた範囲で埋め、**確証が持てない箇所は「（要確認）」と印を付けておく**。生資料は `sources/` に残し、下書きした各ファイルから原典へリンクを張る（後で出典を辿れるように）。

| 下書き先 | 入れる内容 |
|---|---|
| `context/company/overview.md` | 事業概要・主要サービス・収益モデル・強み |
| `context/company/pricing.md` | 料金・単価・プラン |
| `context/clients/{顧客}.md` | 主要顧客ごとのカルテ（任意） |
| `context/members/{名前}.md` | 主要メンバーごとのプロフィール・役割（任意） |
| `context/owner/profile.md` | 代表・本人のプロフィール・判断基準 |
| `reference/writing-style.md` | 営業資料・HPから抽出したブランドトーン・言葉づかい |

各ファイルの冒頭に `updated: YYYY-MM-DD` を入れる（後で「古くなった情報」を検出するため）。

### ③ ユーザーが確認・修正する

下書きした context/ を、**「自社をこう理解しました」とユーザーに要約提示し、誤り・古い箇所・（要確認）印を直してもらう**。ゼロから質問に答えるより、出来上がった下書きを直すほうが速くて正確。修正はその場で context/ に反映する。

### ④ 足りない情報をヒアリングで追記する（繰り返す）

資料だけでは埋まらなかった項目を、**狙って質問する**。以下は context/ が最終的にカバーすべき観点。下書きで埋まったものは飛ばし、空欄・あいまいなものだけを聞く：

- 事業概要（何屋か、創業、ミッション）
- 主要サービス／商品（それぞれ一言で）
- 収益モデル（どう儲けているか、単価感）
- 顧客セグメント（誰が買うか）
- 競合と自社の強み（なぜ選ばれるか）
- 主要メンバーと役割（部下がいる場合）
- 進行中の主要案件・プロジェクト
- ブランドトーン＆禁止表現（対外文書の言葉づかい、使ってはいけない言い回し）

聞いた内容はその都度 context/ に追記する。**この「足りない分を足す」作業は構築時に終わらせきろうとしなくてよい**——運用しながら、新しい事実が出るたびに context/ を継ぎ足していくのが本来の姿（第2部 Step 6 のメモリ振り分けに続く）。

### SSoT原則（事実の置き方）

書き出すときは、以下の原則を守る。これは運用が進んでも一貫して効いてくる。

- **ストック（事実・状態）= 上書き運用** → `context/`。「今こうなっている」を常に最新に保つ。古いと害になる情報
- **フロー（ログ・履歴）= 追記運用** → `work/`・`decisions.md`。時系列で積む。腐らない情報
- **`updated: YYYY-MM-DD`** を context/ の各ファイル冒頭に必ず入れる
- **矛盾したらユーザーに確認**。「いつの情報か分からない事実」は勝手にストックへ書かない
- 新しい事実を聞いたら、その場で該当する context/ ファイルを更新する

### 自社理解度スコア（どこまで分かったかを見える化）

①〜④が一巡したら、AIは「いま自社をどれくらい理解しているか」を採点して提示する。埋まっていない所が一目で分かり、次に足すべき資料が見える：

| 観点 | 状態 |
|---|---|
| 事業概要・サービス | ✓ / △ / ✗ |
| 収益モデル・料金 | ✓ / △ / ✗ |
| 顧客（主要先） | ✓ / △ / ✗ |
| メンバー・体制 | ✓ / △ / ✗ |
| ブランドトーン・禁止表現 | ✓ / △ / ✗ |
| 意思決定の経緯（decisions） | ✓ / △ / ✗ |

✗・△ の観点には「次にこの資料を入れると埋まります」と具体提案する。**100点を目指さず、運用しながら埋めていく**（埋まるほど秘書は賢くなる）。

### 役割別スターターコンテキスト（着手を早める）

ユーザーの役割に応じて、「まず入れる context/」と「最初の3スキル」をテンプレ化してある。該当する役割を参照して着手する：

- [経営者](companion/role-contexts/経営者.md) ／ [営業](companion/role-contexts/営業.md) ／ [マーケティング](companion/role-contexts/マーケティング.md) ／ [カスタマーサポート](companion/role-contexts/カスタマーサポート.md) ／ [人事・採用](companion/role-contexts/人事・採用.md) ／ [経理・財務](companion/role-contexts/経理・財務.md) ／ [総務・バックオフィス](companion/role-contexts/総務・バックオフィス.md) ／ [開発・QA](companion/role-contexts/開発・QA.md)

### 完了条件

- `context/company/sources/` に会社資料（最低でもHP URL）が**先に**入っている
- 資料から `context/company/overview.md` が下書きされ、`updated:` が入っている
- （該当すれば）pricing / clients / members / owner / writing-style が下書きされている
- ユーザーに「自社をどう理解したか」を要約提示し、誤り・（要確認）箇所を修正してもらった
- 資料で埋まらなかった項目をヒアリングで補い、context/ に追記した
- **自社理解度スコア**を提示し、次に足すと効く資料を案内した

## Step 9：コンテキスト運用術【発展】

セッションを賢く・安く保つための運用テクニック。あなた（Claude Code）自身が使い、必要に応じてユーザーにも案内する。

- **`/clear`** — 無関係なタスクに移るときに会話をリセット。文脈の混線を防ぐ
- **`/compact`** — 長時間セッションやスキル連続実行で会話が膨らんだら圧縮する
- **Plan mode** — 大きめの変更前に「まず計画を立てさせ、承認してから実行」する。**この構築指示書を実行するセッション自体がPlan modeの好例**
- **status line** — セッションの状態（モデル・ブランチ・コスト等）を常時表示にカスタムできる
- **モデル使い分け** — 判断・戦略・壁打ち=**Opus** ／ 定型・大量整理=**Sonnet** ／ 軽量定型（日報・ダイジェスト）=**Haiku**。各スキルの frontmatter `model:` で指定。迷ったら対話中はOpus
- **2回ルール** — 2回直しても直らないときは、粘らず `/clear` して最初のプロンプトを書き直す方が速い

最初は `/clear` と `/compact` の2つだけ意識すればよい。長い作業で文脈が膨らんできたら、あなたから `/compact` を提案する。

コンテキスト注入（Step 5）が一巡したら、AIに「この会社専用版のセットアップ手順」を出力させると、2人目以降への展開が速い：

### 付録G：30日ロードマップ

構築が終わったら「30日でどう育てるか」をAIに出力させる。研修第4回の持ち帰り成果物。雛形は [templates/30day-roadmap.md](companion/templates/30day-roadmap.md)：
