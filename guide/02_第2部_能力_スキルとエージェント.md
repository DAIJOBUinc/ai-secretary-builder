## Step 6：Skillsを作成する（厳選カタログ ＋ メモリ近代化）

繰り返し業務をスラッシュコマンド化する。**最初から全部作らない。** よく使う数個を作り、運用しながら足していく。

**どれを初回に作るかは、受講者の役割を聞いて [companion/role-contexts/](companion/role-contexts/) の「鉄板スキル」から選ぶ。** 全員の土台は `/daily-schedule`＋`/mtg-prep` の2つ、その上に役割別の鉄板を乗せる。

### スキルの設計原則

各スキルファイル（`.claude/commands/{名前}.md`）には以下を含める：

- **frontmatter**: `model`（haiku=軽量定型 / sonnet=複雑なタスク / opus=判断・戦略）と `description`（いつ使うか）
- **手順**: 番号付きステップで実行順を明示
- **並列実行の指示**: 独立した情報収集は「並列で実行」と明記（速度が大きく上がる）
- **出力形式**: テンプレートで出力フォーマットを定義
- **出力先**: 保存先パスを明示（例 `work/reports/`）
- **Gotchas（過去にハマったケース）**: ★重要。実際にやらかした失敗を「1行の教訓＋具体ケース＋日付/出典」で書き溜める。スキル内で最も役立つセクションになる（Anthropicのスキル運用で「Gotchasは最高シグナルのcontent」と提唱されている）。最初は空でよく、事故が起きるたびに足す
- **実運用の例を1行添える**: 「実際こう使う」を1行入れると受講者が「自分ごと」にしやすい（下の各スキルの `💡` 参照）

```markdown
## Gotchas（過去にハマったケース）
1. **{1行で言える教訓}**: {何が起きたか／どう判定するか}（{日付・出典}）
```

#### `/daily-schedule`（朝の工程表）— 最優先

```markdown
---
model: haiku
description: 毎朝の15分刻み工程表を作成する（引数で「明日」分も作成可）
---
# 朝の工程表作成

1. カレンダー（コネクタ）から今日の予定を取得（連携済みカレンダー全て）
2. タスク管理ツール（GitHub Issue / Linear / Notion 等、繋いだもの）から進行中・未着手タスクを取得
3. work/daily/ の昨日のログを読み、未完了タスクを確認
4. タスクを分類：緊急度×重要度（アイゼンハワー）／認知負荷（高=設計・分析、中=実装・レビュー、低=事務）
5. 15分刻みの工程表を生成（午前=認知負荷の高い仕事、午後=会議・事務を優先配置）
6. work/daily/ の本日セクションに工程表を書き込む

## 出力形式
YYYY-MM-DD（曜日）の工程表
### 午前 / 午後
| 時間 | タスク | カテゴリ | 認知負荷 |
### 未処理タスク（明日以降）
- [ ]

## Gotchas（過去にハマったケース）
（事故が起きたらここに追記する）
```

> 💡 **実運用例**: 朝イチで回すと、会議の谷間に「重い思考タスク」を寄せ、午後を事務・MTGに固められる。

```markdown
---
model: sonnet
description: MTG前に関連情報を自動収集し、準備資料を生成する
---
# MTG準備

会議前に情報を集め、準備資料を「共有用」と「社内メモ」の2部構成・1ファイルで作る。

## 手順
1. 対象MTGを特定（指定 / 「次のMTG」=カレンダー直近 / 未指定=今日明日一覧から選択）

4. Webリサーチ（サブエージェントに委任）: 相手企業の最新ニュース・業界動向
5. work/reports/mtg-prep/ に保存

## 出力の2部構成
### 【前半】共有セクション（相手に見せてOK）
基本情報 / 前回の振り返り（事実ベースの決定事項のみ）/ アジェンダ案 / 確認事項 / メモ欄
### 【後半】社内メモ（社外秘）
交渉方針 / 料金・コスト / 相手担当者の人物メモ / Webリサーチ結果（出典URL付き）/ 準備チェックリスト

## 重要ルール
- 共有セクションに金額・社内戦略・担当者評価を絶対に入れない

## Gotchas
- 招待メールのアドレス一致だけで「同一人物」と断定しない（同名・別人の取り違え事故あり）
```

```markdown

description: 経営判断や戦略の壁打ち相手。反論・別視点・見落としを出して思考を深める
---
# 壁打ち（スパーリング）

ユーザーの考えに賛成するのが仕事ではない。質を上げるのが仕事。

## 手順
1. ユーザーの主張・前提を3行で要約して確認（ズレてたら直す）

4. 最後に「自分ならこう動く」を1つだけ推奨（評論で終わらせない）

## ルール
- 同意も否定も理由とセット。ふわっとした励ましは禁止

```

### メモリ（近代化）— `/agent-memory` は不要

> ※ 旧バージョンの本書には `/agent-memory` という専用スキルがありましたが、**廃止しました**。

会話中に「覚えておいて」と言われたら、Claude Code 標準のメモリシステムに保存する。仕組み：

- 保存先は `~/.claude/projects/{プロジェクトパス}/memory/`。`MEMORY.md` が索引、各記憶は個別の `.md` ファイル
- 保存するのは**非自明な気づき・フィードバック・方針**だけ（type: user / feedback / project / reference で分類）
- **寿命の長い「事業の事実」は memory/ ではなく context/ に書く**（メモリは最終手段）。「顧客や事業の事実 → context/」「AIへの指示・修正 → memory/」と振り分ける
- 既存の記憶と重複しないか確認し、重複は更新する

専用コマンドを作らなくても、この振り分けルールを CLAUDE.md に書いておけば「覚えておいて」で機能する。

## Step 8：サブエージェント ＆ Agent Team【発展】

### 委任の原則（まずこれ）

1つのセッションで全部やると、コンテキストが汚れて精度が落ちる。**重い作業は別のエージェントに委任して、本体のコンテキストを守る**。

- **委任する基準**: 調査・リサーチ／2つ以上の独立したタスク／3回以上の検索／スキル内の重い処理
- 委任したら**結果を信頼し、自分でやり直さない**（やり直すと委任の意味がない）
- 独立した複数タスクは**並列で投げる**（速度が大きく上がる）

### 単発エージェント vs Agent Team

- **単発エージェント**（`Agent` ツール）: Web調査・テキスト整形・議事録要約・データ取得など、1つの仕事を投げて結果を受け取る。`.claude/agents/{名前}.md` に `model:` と `description:` を書いて定義しておくと、繰り返し使える
- **Agent Team**: 複数の役割が分担して1つの成果物を作る。品質と深さが要る仕事向け。**役割別のチームテンプレ（営業=提案書／開発QA=テスト設計 など）は [companion/role-contexts/](companion/role-contexts/) にある**

### Agent Teamテンプレート

**初回は単発エージェント1〜2個（調査・要約系）から始める。** Agent Team は、受講者の役割テンプレ（[companion/role-contexts/](companion/role-contexts/) の「Agent Team テンプレ」）を1つ選んで作る。下記は横断テンプレ（記事）と、役割テンプレの一覧。

#### テンプレA：記事執筆チーム（`.claude/agents/` ＋ 起動スキル）

```
リサーチャーA（テーゼ→アンチテーゼ→ジンテーゼで論点を立体化）─┐
リサーチャーB（事実・統計・事例を収集）                      ─┼→ 構成設計 → 編集長レビュー → ライター → 編集長 最終レビュー
リサーチャーC（批判的思考・反論・リスク）                    ─┘
```
各リサーチャーは並列実行し、結果を work/content/articles/ に出力。構成→レビュー→執筆→最終レビューの順で品質を上げる。

### Agent TeamのSkills化

作ったチームは `.claude/commands/` のスキルから呼び出せるようにし、コマンド一発で起動できるようにする（例: `/write-article` → 記事執筆チームを起動）。

### 完了条件

## Step 7：定期自動実行（Scheduled Tasks）【発展】

スキルを「手で打つ」から「自動で走る」に進化させる工程。**朝起きたら工程表ができている／夜には日報がまとまっている**という、AI秘書が"勝手に動く"体験の核。

### 仕組み

- 決まった時刻（cron＝定刻に自動実行する仕組み）に、決まったスキル/プロンプトを自動実行するよう登録する
- 登録は **`/schedule` コマンド**（または利用環境のスケジュール機能）で行う。まず使える手段をユーザー環境で確認してから案内する
- 登録時は「**時刻（cron）＋実行するスキル＋結果の通知先**」を指定する（例: 毎朝8時に `/daily-schedule` を実行して結果をSlackに通知）
- 実行形態は2種類：**クラウドのルーティン**（PCを開いていなくても走る）と、**ローカルのスケジュールタスク**（自分のPCで指定時刻に走る）。環境に合う方を選ぶ

### スターターセット（まずこの3つ）

| タイミング | 実行する内容 |
|---|---|
| 平日 朝 | `/daily-schedule` — 工程表を自動生成して通知 |
| 平日 夜 | `/daily-log` — 1日の作業を自動でまとめる |
| 金曜 夕方 | `/weekly-report` — 週次レポートのドラフト作成 |

慣れたら「週次のフォローアップ漏れチェック」「月初のメモリ棚卸し」なども足せる。

### 設計のコツ

- 各定期タスクも、スキルと同じく **手順＋出力先＋Gotchas** を持つ小さな指示書として書く
- 実行結果は通知（Slack DM等）かファイル出力で受け取れるようにする

### Gotchas（定期実行で注意すること）

- **依存データの有無**: 朝タスクが前夜のログに依存する場合、前夜タスクが失敗すると朝も崩れる。各タスクは「データが無くても落ちない」作りにする
- **多重起動**: 同じタスクを二重登録しない。登録済み一覧を先に確認する
- **静かな失敗**: 自動実行は失敗に気づきにくい。「N時間以上実行されていないタスクを検知して通知」する見張りタスクを1本入れておくと安全

### 完了条件

- スターターセットのうち最低1つ（推奨: `/daily-schedule`）が登録され、テスト実行で動く
- 実行結果の受け取り方（通知 or ファイル）が決まっている

## Step 10：Git管理を設定する【発展・任意・スキップ可】

### ヒアリング項目

- Gitを使ったことがあるか（初めて / 聞いたことはある / 使ったことがある）
- 複数PC（会社・自宅）で同じAI秘書環境を使いたいか
- チームでAI秘書の設定を共有する予定があるか

### やる vs やらない（判断ガイド）

| 観点 | Git管理する | しない |
| --- | --- | --- |
| 変更履歴 | 全変更が記録され、壊しても戻せる | 履歴なし。壊したら手動 |
| バックアップ | GitHubにpushすればPC故障でも復元可 | 手動バックアップが必要 |
| 複数端末の同期 | `git pull` で別PCに即再現 | フォルダごとコピー |
| 学習コスト | commit/push の理解が要る（Hooksで自動化すればほぼゼロ） | 追加学習なし |
| セキュリティ | .gitignore を誤ると機密漏洩リスク | 流出リスクが発生しない |

**やるべき人**: 複数PCで作業／壊しても自力で直す自信がない（巻き戻しが安全網）／チームで共有したい人。
**スキップでOK**: まず最小構成で試したい／1台でしか使わない／Gitに時間を使いたくない人。**後からいつでも追加できる。**

### 実行内容（やる場合）

1. プロジェクトフォルダを `git init`
2. `.gitignore` を作成（最低限: `.env` / `*.key` / `credentials/` / `CLAUDE.local.md`（個人設定）/ `context/owner/`（個人情報を含む場合））
3. 初回コミット
4. ユーザーの許可を得て、GitHubに**プライベート**リポジトリを作成しpush

### 発展: git安全装備（自信が付いたら）

事故防止を仕込んでおくと安心：

- **pre-commit フック**でAPIキー・秘密鍵（`AKIA…` / `BEGIN … PRIVATE KEY` / トークン類）をスキャンし、混入時はコミットを止める
- **Bash実行前バリデーション**（Step 4のHook）で破壊系コマンドをブロック
- これらは独立に動き、1つが破られても他が止める多層防御になる

### スキップする場合

最低限、プロジェクトフォルダを定期的にGoogle Drive等へコピーしてバックアップする。**ユーザーが「スキップ」を選んだら、このStepは飛ばして最終確認へ。**
